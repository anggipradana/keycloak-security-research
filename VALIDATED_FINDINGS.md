# Finding 1: CORS OPTIONS Preflight Bypass

## Summary
- CORS `webOrigins` allowlist is not enforced for OPTIONS preflight requests. Any arbitrary origin receives `Access-Control-Allow-Origin` with `Access-Control-Allow-Credentials: true` in preflight responses, enabling cross-origin write-CSRF for admin API and null-origin bypass for all OIDC endpoints.

## Vulnerability Type
- Cross-Origin Resource Sharing (CORS) Misconfiguration

## Affected Component(s)
- Keycloak Server — OIDC/OAuth2 Token Endpoint, UserInfo Endpoint, Admin REST API (`/admin/realms/*`)

## Affected Version(s)
- 26.5.4 (latest stable, confirmed reproducible)

## Keycloak Configuration Context (If Applicable)
- Default installation. Any realm with at least one OIDC client. The `webOrigins` client setting is correctly configured (e.g., `["https://legitimate-app.com"]`) but the enforcement only applies to actual responses, not preflight.

## Detailed Description
Keycloak's CORS filter correctly restricts the `Access-Control-Allow-Origin` (ACAO) header in actual responses (GET/POST) to origins listed in the client's `webOrigins` configuration. However, for `OPTIONS` preflight requests, the filter reflects **any** `Origin` header value with `Access-Control-Allow-Credentials: true`, regardless of whether the origin is in the allowlist.

This creates two attack surfaces:
1. **Write-CSRF via admin API:** If an attacker obtains a valid admin Bearer token (via social engineering, leaked credentials, etc.), JavaScript from `evil.com` can pass the preflight check and send write requests (create user, delete client, etc.) to the admin API. The server-side action completes before the browser blocks reading the response.
2. **null-origin bypass:** The `Origin: null` header (generated by `file://`, `data:` URIs, `<iframe sandbox>`) is also reflected with `credentials: true`, bypassing all `webOrigins` restrictions at the preflight level.

**Key behavior table (client `webOrigins: ["https://legitimate-app.com"]`):**

| Request Type | Origin | ACAO in response | Credentials |
|---|---|---|---|
| `OPTIONS` preflight | `https://evil.com` | `https://evil.com` | `true` |
| `OPTIONS` preflight | `null` | `null` | `true` |
| `POST /token` (actual) | `https://evil.com` | **absent** | — |
| `GET /admin/realms/test/users` (actual) | `https://evil.com` | **absent** | — |

## Impact
- **Authorization policy bypass:** `webOrigins` per-client configuration is not enforced for OPTIONS preflight, meaning administrators who configured origin restrictions have a false sense of security.
- **Write-CSRF for admin API:** If an attacker has obtained a valid admin Bearer token through another vector, they can send destructive write requests (create backdoor users, delete clients, modify realm settings) from any origin. The server-side mutation completes before the browser blocks the response.
- **null-origin universal bypass:** Any OIDC client's webOrigins restriction can be bypassed at the preflight level using `Origin: null` (from sandboxed iframes, file:// URIs, data: URIs).
- **Note:** Actual response reading is correctly blocked for non-allowed origins. Token theft via response reading is NOT possible for correctly-configured clients.

## Recommendations
1. Apply the same `webOrigins` origin validation to `OPTIONS` preflight responses that is already applied to actual responses.
2. Explicitly reject `Origin: null` — never return `Access-Control-Allow-Origin: null` with `credentials: true`.
3. For the admin API, restrict preflight responses to the admin console origin only.

# Finding 2: Unhandled NullPointerException on alg:none JWT

## Summary
- Sending a JWT with `"alg": "none"` to any Bearer-authenticated endpoint triggers an uncaught `NullPointerException`, returning HTTP 500 instead of HTTP 401. This affects all authenticated endpoints including UserInfo, Admin API, and SAML. 100% reproducible, no authentication required.

## Vulnerability Type
- Improper Input Validation / Denial of Service (DoS)

## Affected Component(s)
- Keycloak Server — JWT Validation Pipeline (shared across: OIDC UserInfo Endpoint, Admin REST API, SAML Endpoint)

## Affected Version(s)
- 26.5.4 (latest stable, confirmed reproducible)

## Keycloak Configuration Context (If Applicable)
- Default installation. No special configuration required. Affects every realm.

## Detailed Description
When any Bearer-token-authenticated endpoint receives a JWT whose header declares `"alg": "none"`, Keycloak attempts to retrieve a `SignatureProvider` for the `"none"` algorithm. No such provider is registered, so `getProvider()` returns `null`. The code then calls `.verifier()` on the null reference without a null check, causing an uncaught `NullPointerException`.

The server catches this at the top-level error handler and returns `HTTP 500 Internal Server Error` instead of the correct `HTTP 401 Unauthorized`.

**Verified affected endpoints:**

| Endpoint | Expected | Actual |
|---|---|---|
| `GET /realms/{realm}/protocol/openid-connect/userinfo` | 401 | **500** |
| `GET /admin/realms/{realm}/users` | 401 | **500** |
| `GET /admin/realms/{realm}/clients` | 401 | **500** |

**Reliability:** 100% — tested 20 consecutive requests, all returned 500. Random strings and JWTs with wrong signatures correctly return 401.

**Server log:**
```
ERROR [org.keycloak.services.error.KeycloakErrorHandler]
Uncaught server error: java.lang.NullPointerException:
Cannot invoke "org.keycloak.crypto.SignatureProvider.verifier(String)"
because the return value of "org.keycloak.models.KeycloakSession.getProvider(java.lang.Class, String)" is null
```

## Steps to Reproduce (Proof of Concept - PoC)

**Step 1 — Craft alg:none JWT (no authentication required):**
```bash
# Header: {"alg":"none","typ":"JWT"}
# Payload: {"sub":"attacker","exp":9999999999}
# No signature
ALG_NONE="eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJzdWIiOiJhdHRhY2tlciIsImV4cCI6OTk5OTk5OTk5OX0."
```

**Step 2 — Send to userinfo endpoint:**
```bash
curl -si http://46.101.162.187:8080/realms/test/protocol/openid-connect/userinfo \
  -H "Authorization: Bearer $ALG_NONE"
```

**Expected:**
```http
HTTP/1.1 401 Unauthorized
{"error":"invalid_token","error_description":"..."}
```

**Actual:**
```http
HTTP/1.1 500 Internal Server Error
{"error":"unknown_error","error_description":"For more on this error consult the server log."}
```

**Step 3 — Confirm on admin API:**
```bash
curl -si http://46.101.162.187:8080/admin/realms/test/users \
  -H "Authorization: Bearer $ALG_NONE"
# HTTP/1.1 500 Internal Server Error

curl -si http://46.101.162.187:8080/admin/realms/test/clients \
  -H "Authorization: Bearer $ALG_NONE"
# HTTP/1.1 500 Internal Server Error
```

**Step 4 — Confirm normal JWTs are handled correctly (control test):**
```bash
# Random string → 401 (correct)
curl -si http://46.101.162.187:8080/realms/test/protocol/openid-connect/userinfo \
  -H "Authorization: Bearer this-is-not-a-jwt"
# HTTP/1.1 401 Unauthorized

# Valid JWT structure but wrong signature → 401 (correct)
curl -si http://46.101.162.187:8080/realms/test/protocol/openid-connect/userinfo \
  -H "Authorization: Bearer eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJ0ZXN0In0.invalid"
# HTTP/1.1 401 Unauthorized
```

**Automated PoC script:** `python3 pocs/poc3_alg_none_npe.py`

## Impact
- **Unauthenticated DoS:** Any unauthenticated attacker can flood any Bearer-authenticated endpoint with `alg:none` tokens to force server-side NullPointerExceptions on every request, consuming server resources and polluting error logs.
- **Multiple endpoints affected:** The NPE occurs in the shared JWT validation pipeline, so all Bearer-authenticated endpoints (UserInfo, Admin API, etc.) are vulnerable.
- **Monitoring interference:** HTTP 500 errors from authentication endpoints will trigger false alerts in monitoring systems, masking real attacks.
- **Security code weakness:** The `alg:none` algorithm is not explicitly rejected before provider lookup, indicating a missing security check in the authentication pipeline.

## Recommendations
1. Explicitly reject any token with `"alg": "none"` before attempting provider lookup:
```java
if ("none".equalsIgnoreCase(alg)) {
    throw new ErrorResponseException(OAuthErrorException.INVALID_TOKEN,
        "Algorithm 'none' is not allowed", Response.Status.UNAUTHORIZED);
}
```
2. Add a null check on the provider result to handle any unsupported algorithm gracefully:
```java
SignatureProvider provider = session.getProvider(SignatureProvider.class, alg);
if (provider == null) {
    throw new ErrorResponseException(OAuthErrorException.INVALID_TOKEN,
        "Unsupported algorithm: " + alg, Response.Status.UNAUTHORIZED);
}
```

# Finding 3: Offline Token Persistence After Admin Session Revocation

## Summary
- Offline tokens (issued with `offline_access` scope) survive admin force-logout (`POST /users/{id}/logout`) and push-revocation (`POST /push-revocation`). The admin REST API provides no working endpoint to delete individual offline sessions (DELETE returns 404). This creates a persistent backdoor that survives standard incident response procedures.

## Vulnerability Type
- Improper Session Revocation / Privilege Persistence

## Affected Component(s)
- Keycloak Server — Offline Session Management, Token Revocation, Admin REST API

## Affected Version(s)
- 26.5.4 (latest stable, confirmed reproducible)

## Keycloak Configuration Context (If Applicable)
- Default installation with `offline_access` scope available (default). The vulnerability is worse with default settings:
  - `revokeRefreshToken: false` (default) — offline tokens are not rotated on use
  - `offlineSessionMaxLifespanEnabled: false` (default) — offline sessions never expire

## Detailed Description
When a user obtains an offline token (by requesting `scope=offline_access`), this token is stored as an offline session in the database. Unlike regular sessions, offline sessions:

1. **Survive admin force-logout:** `POST /admin/realms/{realm}/users/{id}/logout` returns 204 (success) but only terminates active sessions, not offline sessions.
2. **Survive push-revocation:** `POST /admin/realms/{realm}/push-revocation` (sets notBefore timestamp) has no effect on offline tokens.
3. **Cannot be individually deleted:** `DELETE /admin/realms/{realm}/users/{id}/offline-sessions/{clientId}` returns HTTP 404.

This means an attacker who obtains an offline token maintains **unlimited, persistent access** regardless of incident response actions, unless the victim's password is explicitly changed.

| Admin Action | Effect on Offline Token |
|---|---|
| `POST /users/{id}/logout` (force logout) | No effect — token still works |
| `POST /push-revocation` (notBefore) | No effect — token still works |
| `DELETE /users/{id}/offline-sessions/{clientId}` | 404 Not Found |
| **Change user password** (only working mitigation) | Invalidates offline token |

## Steps to Reproduce (Proof of Concept - PoC)

## Impact
- **Compromised account recovery is incomplete:** When a security team detects a compromised account and forces logout, the attacker's offline token provides continued access without any indication.
- **Infinite persistence:** With default settings (no token rotation, no session expiry), a single offline token grants permanent access until the password is changed.
- **No admin remediation path:** The admin REST API DELETE endpoint for offline sessions returns 404, meaning admins cannot revoke individual offline sessions programmatically.
- **Bypasses standard incident response:** The notBefore push-revocation mechanism, designed for emergency credential invalidation, has no effect on offline tokens.

## Recommendations
1. Include offline sessions in `POST /users/{id}/logout` — force-logout should revoke both active and offline sessions.
2. Fix `DELETE /users/{id}/offline-sessions/{clientId}` — the endpoint should successfully remove offline sessions (currently returns 404).
3. Apply notBefore policy to offline sessions — `push-revocation` should invalidate offline tokens issued before the notBefore timestamp.
4. Enable `revokeRefreshToken` by default — requiring token rotation on use limits the window of abuse.

# Finding 4: SSRF + Open Redirect via Identity Provider Configuration

## Summary
- Keycloak's Identity Provider subsystem allows three exploitation paths: (A) GET SSRF via `import-config` `fromUrl` parameter for internal network probing, (B) Open redirect via `kc_idp_hint` with a malicious IdP's `authorizationUrl` for credential phishing from a trusted Keycloak domain, and (C) POST SSRF via the IdP `tokenUrl` when the broker processes a callback, enabling POST-capable internal network access.

## Vulnerability Type
- Server-Side Request Forgery (SSRF), Open Redirect / Credential Phishing

## Affected Component(s)
- Keycloak Server — Admin REST API (`/admin/realms/{realm}/identity-provider/import-config`), Auth Endpoint (`/realms/{realm}/protocol/openid-connect/auth?kc_idp_hint=`), Identity Brokering subsystem

## Affected Version(s)
- 26.5.4 (latest stable, confirmed reproducible)

## Keycloak Configuration Context (If Applicable)
- Requires `manage-identity-providers` realm-management role (commonly delegated to development team leads for SSO integration). No non-default realm configuration needed.

## Detailed Description

### Path A — GET SSRF via `import-config` `fromUrl`
The Identity Provider import-config endpoint accepts a JSON body with a `fromUrl` parameter. Keycloak makes a server-side HTTP GET request to the specified URL to download an OIDC discovery document. No URL allowlist or denylist is enforced, allowing:
- Internal network port scanning (timing-based)
- Access to internal HTTP services
- Cloud metadata endpoint access (AWS `169.254.169.254`, GCP, Azure)

### Path B — Open Redirect via `authorizationUrl` + `kc_idp_hint`
An attacker with `manage-identity-providers` role registers a malicious IdP with `authorizationUrl: https://evil.com/auth`. They then craft a phishing URL using Keycloak's standard auth endpoint with `kc_idp_hint`. When a victim clicks:
1. Browser loads legitimate Keycloak URL → Keycloak issues HTTP 303 redirect to `evil.com`
2. Victim's browser lands on attacker server; attacker presents fake login page
3. The initial URL is 100% on the legitimate Keycloak domain, bypassing URL-based phishing filters

### Path C — POST SSRF via `tokenUrl` (Broker Callback)
When the attacker's server returns a fake auth code to Keycloak's broker endpoint, Keycloak makes a **server-side HTTP POST** to the configured `tokenUrl` to exchange the code. The POST body contains `authorization_code`, `redirect_uri`, `client_id`, and `client_secret`. This is a POST-capable SSRF that can target internal services.

- **Internal network probing (Path A):** Attacker can scan internal ports and services from the Keycloak server, including cloud metadata endpoints (`169.254.169.254`) for IAM credential theft.
- **Credential phishing from trusted domain (Path B):** Victims receive a 100% legitimate Keycloak URL that silently redirects to an attacker-controlled server. Bypasses URL-based phishing filters, browser warnings, and user suspicion. Once the malicious IdP is registered, unlimited phishing links can be generated.
- **POST-capable SSRF (Path C):** Keycloak POSTs sensitive data (client credentials, auth codes) to attacker-specified internal URLs, enabling interaction with internal APIs that accept POST requests.
- **Combined attack chain (Path B+C):** A single victim click triggers both the phishing redirect (browser-side) and the POST SSRF (server-side), maximizing impact.

## Recommendations
1. **URL allowlist for `fromUrl`:** Only allow HTTPS URLs; block RFC 1918 private ranges, loopback, and link-local addresses.
2. **Validate `authorizationUrl` and `tokenUrl`:** Enforce URL validation for IdP-configured OAuth URLs — block private/internal addresses.
3. **User confirmation for `kc_idp_hint`:** Show an explicit confirmation page before redirecting to external IdP, rather than immediately issuing HTTP 303.
4. **SSRF protection library:** Implement a shared URL validation utility for all server-side HTTP fetches.

# Finding 5: DCR Trusted Hosts Bypass — Auth Code Interception

## Summary
- Dynamic Client Registration (DCR) "Trusted Hosts" policy only applies to anonymous registrations. Users with the `create-client` realm-management role can register OIDC clients with completely arbitrary `redirect_uris` (e.g., `https://evil.com/steal`) via authenticated DCR, enabling authorization code interception and full token theft for any realm user.

## Vulnerability Type
- Broken Access Control / Privilege Escalation via Client Registration Policy Bypass

## Affected Component(s)
- Keycloak Server — Dynamic Client Registration Endpoint (`/realms/{realm}/clients-registrations/openid-connect`), Client Registration Policy Engine

## Affected Version(s)
- 26.5.4 (latest stable, confirmed reproducible)

## Keycloak Configuration Context (If Applicable)
- Default DCR policy configuration. The `Trusted Hosts` policy exists in the `anonymous` subType but is **absent** from the `authenticated` subType. The `create-client` realm-management role is commonly delegated to application developers.

## Detailed Description
Keycloak's DCR endpoint supports two operation modes:
- **`anonymous`** — No auth required; protected by "Trusted Hosts" policy that validates redirect URIs
- **`authenticated`** — Bearer token required; governed by separate policy set

The `Trusted Hosts` policy (which validates `redirect_uris` against an allowlist) **only exists in the `anonymous` subType**. The `authenticated` subType has no URI restriction policy.

**Policy analysis:**

| Policy | `anonymous` subType | `authenticated` subType |
|---|---|---|
| Trusted Hosts (`client-uris-must-match`) | Enforced | **NOT PRESENT** |
| Allowed Protocol Mapper Types | Enforced | Enforced |
| Allowed Client Scopes | Enforced | Enforced |
| Max Clients Limit | Enforced | Not present |
| Consent Required | Enforced | Not present |

**Verified privilege boundary:**
- `create-client` role → DCR registration with arbitrary URIs: **SUCCEEDS** (201 Created)
- `create-client` role → Admin REST API client creation: **FAILS** (403 Forbidden)
- Anonymous DCR with evil.com redirect: **FAILS** ("Policy 'Trusted Hosts' rejected request")

This means a user with `create-client` role can bypass all URI restrictions via DCR but cannot do so via the admin API — a clear policy enforcement gap.

The registered malicious client is:
- **Immediately enabled** — no admin approval required
- **Fully functional** — can initiate authorization code flows
- **Has a client secret** — attacker can exchange auth codes for tokens

3. Add URI domain validation to the `authenticated` policy set — restrict `redirect_uris` to pre-approved domains.
4. Audit existing DCR clients for unexpected redirect URIs.

# Finding 6: SSRF via Dynamic Client Registration jwks_uri

## Summary
- Users with the `create-client` realm-management role can register OIDC clients via DCR with an arbitrary `jwks_uri`. When a JWT client authentication attempt triggers JWKS fetch, Keycloak makes a server-side HTTP GET request to the attacker-specified URL, enabling blind SSRF for internal network probing.

## Vulnerability Type
- Server-Side Request Forgery (SSRF)

## Affected Component(s)
- Keycloak Server — Dynamic Client Registration (`/realms/{realm}/clients-registrations/openid-connect`), JWT Client Authentication (`private_key_jwt`), JWKS URI fetcher

## Affected Version(s)
- 26.5.4 (latest stable, confirmed reproducible)

## Keycloak Configuration Context (If Applicable)
- Default DCR configuration. The `create-client` realm-management role is commonly delegated to application developers for self-service client registration.

## Detailed Description
Dynamic Client Registration (RFC 7591) supports the `jwks_uri` parameter, allowing a client to specify an external URL from which Keycloak fetches the client's JSON Web Key Set. When a client is registered with `token_endpoint_auth_method: private_key_jwt` and a `jwks_uri`, Keycloak makes a server-side HTTP GET request to that URI during JWT client assertion validation.

**The vulnerability:** No URL validation is performed on `jwks_uri`. A user with `create-client` role can register a client with `jwks_uri: http://127.0.0.1:PORT/path` or any internal address. Triggering a JWT authentication attempt causes Keycloak to fetch the URL server-side.

**Comparison with Finding #4:**

| Aspect | Finding #4 (IdP import-config) | Finding #6 (DCR jwks_uri) |
|---|---|---|
| Required privilege | `manage-identity-providers` | `create-client` |
| HTTP method | GET (Path A), POST (Path C) | GET |
| Trigger | Direct admin API call | DCR registration + JWT auth |
| Request User-Agent | Apache-HttpClient/4.5.14 | Apache-HttpClient/4.5.14 |

## Steps to Reproduce (Proof of Concept - PoC)

**Prerequisites:**
- User with `create-client` realm-management role

**Step 1 — Attacker obtains Bearer token:**
```bash

```

**Step 2 — Start HTTP listener to capture SSRF:**

```bash
    def do_GET(self):
        print(f'SSRF CONFIRMED: {self.command} {self.path}')

            print(f'  {h}: {v}')
        self.send_response(200)

        self.end_headers()
```

```bash
JWT_PAYLOAD=$(python3 -c "
```


```json

```
The error confirms Keycloak attempted to fetch the JWKS (and the empty `{"keys":[]}` had no matching key), proving the server-side HTTP request was made.

## Impact
- **Internal network reconnaissance:** Attacker can probe internal hosts and port availability by registering clients with different `jwks_uri` targets and observing timing/error differences.
- **Cloud metadata access:** In AWS/GCP/Azure deployments, the SSRF target can be the instance metadata service (`169.254.169.254`) to retrieve cloud IAM credentials.
- **Lower privilege than Finding #4:** Uses `create-client` role instead of `manage-identity-providers`, broadening the attack surface to more users.
- **Combined with Finding #5:** The same `create-client` role enables both token theft (Finding #5) and internal network scanning (Finding #6).

## Recommendations
1. Validate `jwks_uri` against an allowlist — only allow HTTPS URLs from trusted domains; block private IP ranges (RFC 1918), loopback, and link-local addresses.
2. Apply the same SSRF protections as Finding #4 — implement a shared URL validation utility for all server-side HTTP fetches.
3. Require elevated privileges for `private_key_jwt` with external `jwks_uri` — add a DCR policy requiring `manage-clients` role for this authentication method.
