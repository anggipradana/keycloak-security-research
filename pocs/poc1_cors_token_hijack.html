Technical Analysis

VULNERABILITY:
  Endpoint: POST /realms/{realm}/protocol/openid-connect/token
  Header:   Origin: https://evil.com  (NOT in client's webOrigins)
  Expected: No Access-Control-Allow-Origin header (browser blocks response)
  Actual:   Access-Control-Allow-Origin: https://evil.com
            Access-Control-Allow-Credentials: true
            → Browser allows cross-origin reads!

AFFECTED ENDPOINTS (all return ACAO: <any-origin>):
  - /realms/{realm}/protocol/openid-connect/token
  - /realms/{realm}/protocol/openid-connect/userinfo
  - /realms/{realm}/account
  - /admin/realms/{realm}/users (admin API)

ROOT CAUSE:
  Keycloak's CorsFilter reflects the incoming Origin header without
  checking it against the client's configured webOrigins whitelist.
  The webOrigins per-client setting has zero enforcement effect.

ATTACK SCENARIOS:
  1. Phishing page collects credentials → cross-origin token fetch →
     access_token stolen (no CSP/SOP protection)
  2. Malicious npm package in web app → XHR to KC token endpoint →
     read token response
  3. null-origin attack via sandboxed
